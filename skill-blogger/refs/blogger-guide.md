# 블로거 스타일 가이드 (상세)

## 1. 독자 타겟 & 톤앤매너

### 타겟 독자
- **주요**: 해당 분야 입문~중급 개발자/기술자
- **부차**: 기술 트렌드에 관심 있는 비개발자, 의사결정자

### 문체 원칙
- **친근하되 전문적**: "~합니다" 체, 과도한 존댓말 지양
- **능동형 문장**: "~됩니다" 대신 "~합니다", "~할 수 있습니다"
- **구체적 표현**: 추상적 서술 지양, 예시와 비유 적극 활용
- **간결함**: 한 문장 40자 이하 권장, 문단 5줄 이하

## 2. 제목 작성 규칙

### SEO 최적화 제목 패턴
```
[기술명] + [핵심 키워드] + [독자 이익]
예) "Spring Boot 3.x로 REST API 10분 만에 구축하기"
    "Docker Compose 완전 가이드: 로컬 개발환경 표준화"
    "Redis 캐싱 전략: 서비스 응답속도 3배 개선하기"
```

### 피해야 할 제목 패턴
- 내부 프로젝트명 포함: "XX프로젝트 기술 가이드" (❌)
- 너무 모호한 제목: "기술 정리" (❌)
- 기밀 정보 암시: "내부 시스템 아키텍처" (❌)

## 3. 내부 기밀 vs 공개 가능 판단 기준

### 절대 공개 금지 (RED)
| 항목 | 예시 |
|------|------|
| 내부 시스템명/코드명 | "프로젝트 ALPHA", "내부망 서버 XX-01" |
| 독점 알고리즘 상세 | 자체 개발 핵심 로직, 특허 출원 전 기술 |
| 인프라 정보 | IP 주소, 도메인, 포트, 보안 그룹 설정 |
| 고객/파트너 정보 | 고객사명, 계약 조건, SLA 수치 |
| 인증 정보 | API 키, 비밀번호, 토큰, 인증서 |
| 미발표 로드맵 | 출시 예정 기능, 전략적 계획 |

### 공개 가능 (GREEN)
| 항목 | 예시 |
|------|------|
| 범용 기술 개념 | Spring Boot, Kubernetes, GraphQL 원리 |
| 오픈소스 활용법 | GitHub에 공개된 라이브러리 사용 방법 |
| 업계 표준 패턴 | MSA 아키텍처, CQRS, Event Sourcing |
| 공식 문서 요약 | AWS 공식 문서, RFC 표준 설명 |
| 일반 코드 예제 | 튜토리얼 수준의 범용 코드 |

### 주의 필요 (YELLOW) → 일반화 후 사용
| 항목 | 처리 방법 |
|------|----------|
| 특정 기술 스택 조합 | 범용적 관점으로 재서술 |
| 성능 수치 | "약 N배 개선" 수준으로 일반화 |
| 팀 규모/조직 구조 | 일반적 팀 구성으로 추상화 |

## 4. 섹션 구성 패턴

### 기본 포스팅 구조
```
1. 요약 (Summary) - 3~5문장
2. 들어가며 - 문제 제기, 배경
3. 핵심 개념 설명 - 이론/원리
4. 실습/예제 - 코드 포함
5. 심화/응용 - 실전 적용
6. 마무리 - 요약 + 다음 단계
7. 참고 자료
```

### 내용 길이 기준
- **단문 포스팅**: 1,500~2,500자 (개념 소개, 팁)
- **표준 포스팅**: 2,500~4,000자 (기술 설명, 튜토리얼)
- **심층 포스팅**: 4,000~6,000자 (아키텍처, 비교 분석)

## 5. 코드블럭 작성 규칙

### 필수 규칙
- 언어 명시 필수: ` ```java `, ` ```python `, ` ```bash ` 등
- 실행 가능한 예제 선호 (단, 내부 의존성 제거 후)
- 주석으로 핵심 포인트 설명
- 코드 내 내부 정보 제거: 내부 URL, 시스템명, 키값 → 플레이스홀더 대체

### 플레이스홀더 처리 예시
```yaml
# 원본 (내부 정보 포함 - 공개 금지)
database:
  url: jdbc:mysql://internal-db.company.com:3306/prod_db
  username: admin
  password: secret123

# 공개용 (플레이스홀더 대체)
database:
  url: jdbc:mysql://${DB_HOST}:${DB_PORT}/${DB_NAME}
  username: ${DB_USERNAME}
  password: ${DB_PASSWORD}
```

## 6. 이미지 처리 가이드

### 이미지 저장 경로
```
{post-title}/img/{post-title}/{이미지파일명}
```

### 마크다운 이미지 삽입
```markdown
![이미지 설명](img/{post-title}/{파일명}.png)
```

### 이미지 내 기밀 정보 처리
- 스크린샷에 내부 시스템명, IP, 도메인 → 모자이크/블러 처리 필요
- 직접 편집 불가 시 → 이미지 설명 텍스트로 대체
- 다이어그램의 내부 컴포넌트명 → 범용 명칭으로 대체
